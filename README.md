# Реализация

## Синхронизация между процессами

1. Сначала я понял задание так, что 256 байт - размер только буффера данных, механизмы синхронизации считаются отдельно.

    Поэтому реализовал алгоритм синхронизации, использующий две conditional variable, ограничивающие только служебные поля структуры данных(флаг готовности и размер записанных данных).

    Преимущество алгоритма заключается в том, что оба процесса могут редактировать разные части буффера одновременно.

    Однако для буффера столь малого размера (256 байт) преимущество невилируется полностью накладными расходами на лишний вызов conditional_variable::wait ().

2. Затем я уточнил, что 256 байт - это размер всей структуры данных, включая механизмы синхронизации. Поэтому от идеи с двумя conditional_variable отказался окончательно, выбрал следующий алгоритм:


    Принцип работы:

    ```
        Блокировка мьютекса;
        Если данные еще не готовы - ожидание готовности данных через conditional_variable;
        Работа с общей памятью;
        Отправка сигнала об освобождении conditional_variable;
    ```
3. Мысли про оптимизацию:

    В данном случае можно было бы обойтись одним мьютексом, без conditional variable, тогда бы удалось оставить больше свободного места на полезную нагрузку при передаче данных. 

    Однако нагрузка на CPU бы увеличилась из-за бесконечной проверки мьютекса при ожидании новых данных. Поэтому было принято решение оставить алгоритм из П.2.

## Алгоритм сжатия данных ##

1. Сначала был выбран и реализован алгоритм сжатия данных через библиотеку ZFP: ([API ZFP](https://github.com/LLNL/zfp)). 

    Его смысл состоит в том, что алгоритм группирует соседние числа при сжатии. Поэтому для АБСОЛЮТНО случайных чисел алгоритм работает плохо.

    При чтении исходных данных с разделителем \n между числами(См. описание в самом начале. Долгое время при выполнении задания я работал с неправильными данными) - набор чисел получается абсолютно случайным:

    Первые 5 чисел файла при неправильном чтении:

    ```
    0.6232919692993164
    -2.8527379072329495e-06
    -1.689097228340118e+32
    1.7501359519597748e-13
    0.13029953837394714
    ```

    Для такого набора потери алгоритма ZFP являются критическими, поэтому было принято решение от него отказаться. 
    
    После того, как нашел ошибку с неверным форматом входных данных - решил не возвращаться к алгоритму ZFP, т.к. алгоритм из П. 2 явлется более надежным и не зависит от структуры входных данных. 

2. Было решено использовать следующий алгоритм сжатия и передачи данных:

    ```
    Для каждого числа:
        На стороне отправителя путем побитового сдвига обрезаются 16 младших бит мантиссы;
        Два оставшихся байта копируются в буффер для передачи на приемник.
        На приемнике значение числа восстанавливается путем обратного побитового сдвига на 16 бит.
    ```

    Преимущества алгоритма:

    * Каждое число сжимается по отдельости, результат сжатия не зависит от структуры данных, алгоритм подходит для обработки случайных/непредсказуемых данных.

    * Изменением числа обрезаемых бит можно контролировать соотношение процента сжатия/процента потери данных.

### Запуск работы

```bash
# Запустить Producer:

./build/Producer <compress_mode> <input_file_path>

# Где compress_mode:
#       compress_enable - сжатие включено
#       compress_disable - сжатие выключено
#       input_file_path - Путь до файла с входными данными

# На экран будет выведен ID разделенной памяти.
# Его нужно  скопировать и передать в аргумент при запуске Consumer.


# Запустить Consumer:
./build/Consumer <compress_mode> <ID> <output_file_path>

#       compress_mode    -      Режим сжатия, должен быть такой же как у Producer
#       ID               -      Скопированный ID памяти Producer
#       output_file_path -  Путь для сохранения выходного файла.

# Для начала передачи данных ввести клавишу Enter на ввод в Producer.

```

Размер буффера с данными вычислен путем вычисления смещения самого буфера (Макрос offsetof) относительно начала структуры.

Полученный размер задан константой в коде. 

На других платформах размер может отличаться из-за особенностей типов pthread_cond_t и pthread_mutex_t.
## Заключение про алгоритм сжатия

Алгоритм, реализованный сейчас - по сути не алгоритм сжатия, а алгоритм обрезания младших битов. 

Было принято решение отказаться от алгоритма реального сжатия из-за того, что при действительно случайных данных (например, если на соседних местах встречаются значения, отличающиеся на десятки порядков) - алгоритмы сжатия работают непредсказуемо.

К тому же в алгоритме ZFP, API которого изучил в ходе работы, есть возможность контролировать только один из двух параметров - или процент потерь, или процент сжатия. Метод обрезания битов дает контроль над обоими параметрами.
